# Future Admin Guide (MVP 이후 Admin 대비)

## 목적
MVP(게시판/유저/업무관리)가 어느 정도 완성되면 운영·테스트를 위해 Admin(관리자) 기능이 필요해질 가능성이 높다.
Admin을 나중에 빠르게 만들기 위해, 지금부터 아래 3가지 원칙을 지킨다.

---

## 원칙 1) 공통 코드 경계 만들기
### 왜?
나중에 Admin이 생기면 Web/Admin이 공통으로 쓰는 코드(컴포넌트, 유틸, 타입, API 서비스)가 생긴다.
지금부터 공통 코드를 "페이지 전용 코드"와 분리해두면, 나중에 packages로 분리하거나 Admin 앱에 재사용하기 쉽다.

### 지금부터의 규칙
- 페이지(라우트) 전용 코드는 각 페이지 근처에 둔다.
- 여러 곳에서 재사용 가능한 것은 공용 폴더로 이동한다.

### 권장 예시
- 공용(재사용 가능):
  - `src/components/**` (UI 컴포넌트/공용 컴포넌트)
  - `src/lib/**` (유틸, 타입, 상수)
  - `src/services/**` (API 호출/서비스 로직)
- 페이지 전용:
  - `src/app/**` 내부에서 해당 라우트에만 쓰는 파일

### 체크리스트
- [ ] 새 컴포넌트를 만들 때 “Admin에서도 쓸 가능성”을 10초만 생각한다.
- [ ] 재사용 가능하면 공용 폴더에 둔다.
- [ ] 페이지 전용이면 라우트 폴더 안에 둔다.

---

## 원칙 2) 환경변수/설정 파일 정리
### 왜?
Admin이 생기면 Web/Admin이 같은 Supabase 프로젝트(또는 동일한 설정)를 공유할 수 있다.
설정이 흩어져 있으면 Admin 만들 때 환경설정/키 관리가 꼬인다.

### 지금부터의 규칙
- Supabase 클라이언트 생성 코드는 한 군데에서만 만든다.
- 환경변수(.env)는 의미가 드러나게 유지한다.
- "공유 값"과 "서버 전용 값"의 구분을 유지한다.

### 체크리스트
- [ ] Supabase client 생성 파일이 단일 위치에 있는가?
- [ ] env 키들이 어디서 쓰이는지 문서화/주석이 있는가?
- [ ] 서버 전용 키(예: service role)는 클라이언트 번들에 들어가지 않도록 분리할 계획이 있는가?

## 원칙 3) SQL 스크립트는 supabase/ 아래로 정리
### 왜?
운영 중에는 RLS/뷰/함수/트리거/정책을 자주 수정하게 된다.
SQL 파일이 루트에 흩어져 있으면 유지보수가 어렵고, 누가 어떤 스크립트를 언제 실행했는지 추적이 힘들다.

### 지금부터의 규칙
- 루트에 SQL 파일을 새로 만들지 않는다.
- DB 관련 SQL은 `supabase/` 아래에 모아둔다.
- 용도에 따라 폴더를 나눈다.

### 권장 구조 (가벼운 정리 버전)
- `supabase/sql/schema/` : 테이블/뷰/함수/트리거 등 스키마 정의
- `supabase/sql/policies/` : RLS 정책
- `supabase/sql/fixes/` : 일회성 수정 스크립트(파일명에 날짜/이슈번호 포함)

### 체크리스트
- [ ] 새 SQL은 항상 `supabase/sql/**` 아래에 저장하는가?
- [ ] 실행 순서/주의사항이 필요한 SQL은 상단 주석으로 남겼는가?

## Admin은 어떻게 시작할까? (권장)
- 초기에는 별도 앱을 만들지 말고, **같은 Next.js 앱에서 `/admin` 라우트로 “미니 어드민”부터 시작**한다.
- MVP 이후 운영/테스트에 필요한 최소 기능부터 제공한다.
- Admin이 커지고 Web/Admin 분리가 필요해지면, 그때 모노레포 전환을 검토한다.

---

## 미니 어드민(초기) 우선순위 예시
- 유저 목록 + 상태 변경(정지/권한)
- 게시글/댓글 삭제 또는 숨김
- 공지/배너 등록
- 신고 리스트 + 처리 상태 변경
- 테스트 데이터/업무 상태 수동 수정
